# Earth Online 项目深度分析与发展规划

> 日期：2026-04-24
> 作者：Kiro
> 基于对项目完整代码库、文档体系、EverOS v1 技术报告的深度阅读

---

## 一、项目本质理解

Earth Online 不是一个任务管理工具。它是一个**记忆驱动的行为陪伴系统**。

市面上的任务工具（Todoist、Things、TickTick）解决的是"记录该做什么"的问题。Earth Online 解决的是一个更深层的问题：**当一个人失去动力、中断节奏、不知道从哪里重新开始的时候，系统能不能帮他接上？**

这个区别决定了整个产品的技术架构。传统任务工具的核心是数据库里的 CRUD——创建任务、标记完成、删除。Earth Online 的核心是**记忆层**——系统记住你做过什么、你的节奏是什么样的、你在哪里断掉的、你上次恢复用了什么方法。然后基于这些记忆，在你下次打开 App 的时候，给你一个有温度的、具体的、可执行的下一步。

从代码结构看，这个理解体现在几个关键设计决策上：

1. `guide_memory.ts` 不是一个简单的数据查询层。它做了六件事：查询任务状态、拉取每日日志、获取用户画像、检索近期对话、调用 EverMemOS 做语义检索、构建行为信号。这六个数据源被压缩成一个 `packed_context`，注入到每一次 LLM 调用里。这意味着系统的每一句话都是"带着记忆说的"。

2. `guide_ai.ts` 里的 prompt 设计不是通用的 chatbot prompt。它明确要求 LLM "引用至少一条历史事实"、"禁止模板化安慰"、"title 必须是具体可执行的小动作"。这些约束把 LLM 从一个泛泛的聊天机器人变成了一个"知道你最近在干什么"的陪伴者。

3. `agent_planner.ts` 的意图识别是基于关键词的规则匹配，不是 LLM 推理。这是一个务实的选择——对于"创建任务"、"打开统计页"这类明确意图，规则匹配比 LLM 更快更可靠。LLM 只在 freeform 路径（自由聊天）时才被调用。

4. 整个系统的错误处理哲学是"降级而不是崩溃"。EverMemOS 不可用？用本地数据继续。LLM 调用失败？用 fallback 文案。图像生成失败？降级到 Pollinations。这种设计让产品在基础设施不稳定的情况下仍然可用。

## 二、记忆系统的当前状态

经过这次升级（v1.4.0 → v1.5.0），记忆系统已经从"隐形基础设施"变成了一个有四个触点的完整闭环：

**写入侧（4个入口）：**
- 用户完成任务 → `sync-user-memory` 写入 EverMemOS
- Agent 执行目标 → `syncAgentEventToMemory` 写入 `agent_goal`
- Agent 工具执行完成 → 写入 `agent_tool_result`
- Agent run 终态 → 写入 `agent_run_complete`

**检索侧（3种模式）：**
- Hybrid search：向量相似度 + 关键词匹配，用于 Guide 对话
- Agentic search：上下文感知检索，用于 Agent 规划
- Cross-task search：跨任务关联检索，用于发现任务间的历史联系

**展示侧（2个界面）：**
- Guide 对话气泡：可展开的记忆引用标签，按来源类型分类
- 独立记忆面板：搜索、浏览、按类型分类的记忆卡片

**主动侧（1个巡逻）：**
- `memory-patrol`：检测断签、任务搁置、长时间沉默，主动推送提醒

但这只是记忆系统能做的事情的冰山一角。

## 三、记忆能做的其他事情

### 3.1 记忆衰减与知识提取

EverOS v1 技术报告里描述了一个 Ebbinghaus 衰减模型：Easy 记忆 24 小时半衰期，Medium 72 小时，Hard 168 小时。低于 30% 保留阈值的记忆被归档，但提取的知识模式被保留。

当前系统没有实现这个机制。所有记忆都是平等的，没有衰减权重。这意味着：
- 三个月前完成的一个临时任务和昨天完成的核心目标在检索时权重相同
- 记忆会无限积累，检索质量会随时间下降
- 用户无法区分"我现在的状态"和"我三个月前的状态"

**建议实现：**
在 `guide_memory.ts` 的 `gatherGuideMemoryBundle` 里，对检索结果按时间加权。近 7 天的记忆权重 1.0，7-30 天 0.6，30-90 天 0.3，90 天以上 0.1。同时调用 EverOS 的 `flush` API 定期触发知识提取，把"你每周五都会做复盘"这类模式从具体记忆中抽象出来，存为 `semantic_memory`。

### 3.2 记忆画像演化

当前的记忆画像（`guide-portrait-generate`）是一次性的——每次生成都是基于当前记忆快照的静态图片。但记忆画像的真正价值在于**演化**：用户应该能看到自己从一个月前到现在的变化。

**建议实现：**
- 在 `guide_portraits` 表里增加 `epoch` 字段，标记画像属于哪个时间段
- 每周自动生成一张画像（通过 pg_cron 触发 `guide-portrait-generate`）
- 前端增加"画像时间线"视图，用户可以左右滑动看到自己的演化轨迹
- 画像的 prompt 里注入上一张画像的 summary，让 LLM 能描述"和上次相比，你的变化是..."

### 3.3 记忆驱动的任务推荐

当前的任务推荐（`GuideSuggestedTask`）是基于当前对话上下文的一次性建议。但如果结合记忆，可以做到更精准的推荐：

- 用户上周完成了"健身 30 分钟"三次 → 推荐"本周目标：健身 4 次"
- 用户连续三天在晚上 11 点后完成任务 → 推荐"恢复支线：今晚 10 点前停止工作"
- 用户创建了"学英语"任务但一周没动 → 推荐"把学英语拆成 15 分钟的小任务"

**建议实现：**
新增一个 `memory-recommender` Edge Function，在 `guide-bootstrap` 时调用。它从 EverMemOS 检索用户最近 30 天的行为模式，用 LLM 生成 2-3 个个性化任务推荐，注入到 `GuideBootstrapResult` 里。前端在首页显示为"小忆建议"卡片。

### 3.4 跨用户记忆模式（匿名化）

EverOS v1 的 Group scope 允许多个用户共享一个记忆空间。这意味着可以构建一个匿名化的"群体智慧"层：

- 统计所有用户中，哪些类型的任务最容易被放弃
- 发现"连续打卡 7 天后断签"是一个普遍模式，提前在第 6 天推送鼓励
- 把成功恢复节奏的用户的行为模式提取出来，推荐给正在断签的用户

**建议实现：**
创建一个 `group_id: "earth-online-collective"` 的共享记忆空间。每当用户完成一个里程碑（连续 7 天、首次清盘、从断签中恢复），写入一条匿名化的 `semantic_memory`。`memory-patrol` 在检测到用户断签时，从这个共享空间检索"其他人是怎么恢复的"，注入到推送消息里。

### 3.5 记忆可编辑性

当前用户无法编辑或删除自己的记忆。这是一个产品缺陷——用户应该有权决定系统记住什么、忘掉什么。

**建议实现：**
- 记忆面板增加"标记重要"和"忘掉这条"操作
- "标记重要"调用 EverOS 的 sender API 更新 metadata，增加 `pinned: true`
- "忘掉这条"调用 `memories.delete()` 或标记 `sourceStatus: "muted"`
- `guide_memory.ts` 在检索时过滤掉 `muted` 状态的记忆

## 四、功能发展方向

### 4.1 短期（1-2 周）

**微信记忆同步**
当前微信 webhook 只把消息解析成任务。但微信消息本身就是一种记忆——用户发的"今天好累"、"终于搞定了"这些碎片化表达，包含了情绪和状态信息。

建议在 `wechat-webhook` 里，除了解析任务，同时把原始消息写入 EverMemOS 作为 `dialog_event`。这样 Guide 在对话时能引用"你昨天在微信里说好累"，让陪伴感更真实。

**夜间反思记忆闭环**
`guide-night-reflection` 已经能生成夜间复盘，但复盘结果没有写回记忆。建议在反思完成后，把 `opening` + `follow_up_question` + 用户的回答一起写入 EverMemOS，形成"日记"类型的记忆。这样下次反思时能引用"上周你说想早点睡"。

### 4.2 中期（1-2 月）

**Sender 身份追踪**
EverOS v1 的 Sender API 允许追踪"谁写了什么"。当前所有记忆都是匿名写入的，无法区分是用户主动上传的、Guide 自动生成的、还是 Agent 执行产生的。

建议为每个写入源注册一个 Sender：
- `sender: "user-manual"` — 用户手动上传
- `sender: "guide-assistant"` — Guide 对话产生
- `sender: "agent-runtime"` — Agent 执行产生
- `sender: "patrol-nudge"` — 主动推送产生
- `sender: "wechat-webhook"` — 微信消息产生

这样在记忆面板里可以按来源过滤，用户能清楚地看到"哪些是我自己说的，哪些是 AI 生成的"。

**习惯链检测**
基于记忆中的时间序列数据，检测用户正在形成的习惯链。比如：
- 连续 5 天在早上 8 点完成第一个任务 → 检测到"早起习惯链"
- 每周三都会做复盘 → 检测到"周三复盘习惯"
- 每次完成高强度任务后都会做恢复任务 → 检测到"推进-恢复节奏"

检测到习惯链后，在 Guide 对话中主动提及："你已经连续 5 天早起了，这个习惯正在形成。"如果习惯链断裂，在 `memory-patrol` 中触发提醒。

### 4.3 长期（3-6 月）

**多 Agent 协作记忆**
当前只有一个 Agent（Guide/小忆）。但 EverOS v1 的 Agent scope 支持多个 Agent 各自拥有独立记忆空间。可以引入专业化的 Agent：
- **效率 Agent**：专注于任务拆分、时间管理、优先级排序
- **恢复 Agent**：专注于休息、运动、情绪调节
- **成长 Agent**：专注于学习、复盘、技能提升

每个 Agent 有自己的记忆空间（`agent_id`），但可以通过 Group scope 共享用户的基础行为数据。Observer Agent（类似 EverOS 技术报告中的 Panopticon 拓扑）可以跨 Agent 检索，协调多个 Agent 的建议不冲突。

**记忆驱动的游戏化演化**
当前的游戏化（XP、金币、成就）是静态的——完成任务得 XP，XP 够了升级。但如果结合记忆，游戏化可以变得动态：
- 基于用户的行为模式动态调整 XP 倍率（连续推进时倍率上升，断签后倍率下降但恢复任务给额外 XP）
- 成就系统基于记忆模式解锁（"连续 30 天有记忆记录" → 解锁"记忆守护者"成就）
- 每日事件（`guide_daily_events`）基于记忆生成个性化挑战，而不是随机生成

**语音记忆**
当前所有记忆都是文本。但 Flutter 已经集成了 `speech_to_text` 包。可以让用户通过语音输入记忆——"今天完成了三个任务，感觉还不错"。语音转文字后写入 EverMemOS，同时保留语音文件到 Supabase Storage。

这样记忆面板里可以播放原始语音，让"回忆"更有沉浸感。

## 五、技术架构演进建议

### 5.1 记忆层抽象

当前 `EverMemOSClient` 直接在 `guide_memory.ts`、`agent_engine.ts`、`memory-patrol` 等多个地方被实例化。建议抽象出一个 `MemoryLayer` 接口：

```typescript
interface MemoryLayer {
  write(userId: string, event: MemoryEvent): Promise<void>;
  search(userId: string, query: string, opts: SearchOpts): Promise<MemoryItem[]>;
  agenticSearch(userId: string, query: string): Promise<MemoryItem[]>;
  flush(userId: string): Promise<void>;
}
```

这样如果未来需要切换记忆后端（比如从 EverMemOS 切到自建的向量数据库），只需要实现一个新的 `MemoryLayer`，不需要改动所有调用方。

### 5.2 前端记忆缓存

当前每次打开记忆面板都要调用 EverMemOS API。建议在前端加一层本地缓存：
- 用 `shared_preferences` 缓存最近 50 条记忆
- 搜索结果缓存 5 分钟
- 离线时显示缓存数据，联网后自动刷新

### 5.3 记忆写入队列

当前 `syncAgentEventToMemory` 是 fire-and-forget 的异步调用。如果网络不稳定，记忆会丢失。建议在 Supabase 里加一个 `memory_write_queue` 表，先写入队列，再由后台任务批量同步到 EverMemOS。失败的写入可以重试。

## 六、前景判断

Earth Online 的核心竞争力不在于任务管理本身——这个赛道已经非常拥挤。它的竞争力在于**记忆层**。

当前市面上没有一个效率工具真正做到了"记住用户"。Todoist 不知道你上周的节奏是什么样的。Things 不知道你每次断签后是怎么恢复的。Notion 不知道你三个月前设定的目标和现在的行动之间有什么关联。

Earth Online 通过 EverMemOS 的三层记忆架构（Personal / Group / Agent），有机会成为第一个真正"有记忆"的效率工具。这个记忆不是简单的历史记录——它是经过语义理解、模式提取、衰减过滤后的"活的记忆"，能够影响系统的每一次决策。

从商业角度看，记忆层也是最强的护城河。用户在系统里积累的记忆越多，迁移成本越高。这不是 lock-in——而是用户真正的数据资产。如果做好记忆的可导出性（比如支持导出为 Markdown 日记），用户会更愿意持续使用，因为他们知道自己的数据是安全的。

从技术角度看，EverOS v1 的物理隔离模型（每个 API key 对应独立的记忆空间）解决了多用户场景下的数据隔离问题。这意味着 Earth Online 可以安全地扩展到多用户、多 Agent 场景，而不用担心记忆交叉污染。

最大的风险是**记忆质量**。如果写入的记忆太多、太杂、太噪声，检索质量会下降，LLM 生成的回复会变得不相关。这就是为什么记忆衰减、知识提取、用户可编辑性这三个功能如此重要——它们是保证记忆质量的三道防线。

## 七、总结

Earth Online 的本质是一个实验：**如果一个效率工具有了持久记忆，它能不能从"工具"变成"陪伴者"？**

从代码库的现状来看，这个实验已经有了一个可工作的原型。记忆的写入、检索、展示、主动推送四个环节都已经打通。接下来的关键是：

1. 提高记忆质量（衰减、提取、可编辑）
2. 让记忆的作用更可感知（画像演化、习惯链检测、个性化推荐）
3. 扩展记忆的来源（微信消息、语音输入、夜间反思）
4. 构建记忆的护城河（多 Agent 协作、群体智慧、Sender 追踪）

这不是一个需要一次性完成的大工程。每一个小功能都可以独立上线、独立验证。重要的是保持"记忆优先"的产品思维——每做一个新功能，先问自己：这个功能产生的数据，能不能变成记忆？这个功能的决策，能不能被记忆驱动？

如果答案是"能"，那就是对的方向。
