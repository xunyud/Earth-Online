# 系统商城设计文档

> 日期：2026-03-24
> 执行者：Codex

## 1. 背景

当前项目已经具备奖励商城、背包、系统商品字段、`buy_reward` RPC 与奖励任务落地链路，但商城页面实际上只展示了用户自定义奖励，系统商品能力没有完整接入到前端主流程。

本次目标是在不新建平行模块的前提下，把现有系统商品能力正式接入商城页面，并提供一批适合日常使用的初始系统商品。

## 2. 目标

- 在现有奖励商城中新增“系统商城”展示区域。
- 左右分栏展示：
  - 左侧为固定系统商品。
  - 右侧为用户自定义奖励。
- 系统商品不可编辑、不可删除。
- 系统商品兑换后先进入背包，再由用户手动“使用”。
- 使用后沿用现有逻辑生成只读奖励任务。
- 提供一批内置初始商品，价格最低为 `1` 金币。

## 3. 非目标

- 不重构现有奖励、背包、任务结算的整体架构。
- 不新增独立的系统商城表。
- 不改变自定义奖励的创建与删除规则。
- 不引入新的远程配置系统或前端兜底商品源。

## 4. 现有实现基础

### 4.1 数据层

- `rewards` 表已支持：
  - `is_system`
  - `category`
  - `description`
  - `effect_type`
  - `effect_value`
  - `is_active`
- `inventory` 表已支持：
  - `reward_id`
  - `effect_type`
  - `effect_value`
  - `is_used`
  - `is_equipped`
- `buy_reward(r_id uuid)` 已支持：
  - 原子扣金币
  - 写入背包
  - 继承商品效果字段

### 4.2 Flutter 层

- `RewardController` 已区分：
  - `_customRewards`
  - `_systemRewards`
- `loadRewards()` 已能同时拉取系统商品和用户商品。
- `useItem()` 已支持：
  - 对 `effectType == null` 的奖励创建只读奖励任务
  - 对系统道具保留背包使用逻辑
- 当前缺口在于：
  - `RewardShopPage` 只渲染了自定义奖励列表
  - 没有把系统商品真正展示到商城 UI

## 5. 核心决策

### 5.1 数据方案

采用“数据库驱动，前端双栏展示”方案。

- 系统商品继续存放在 `rewards` 表中。
- 使用 `is_system = true` 区分系统商品。
- 使用现有 `buy_reward` RPC 完成兑换。
- 不新增新的商品来源，不在前端写死主数据作为正式来源。

原因：

- 与现有 `RewardController`、`inventory`、`buy_reward`、背包链路完全兼容。
- 后续新增或调整系统商品时，只需要继续维护数据库种子数据。
- 避免前端和数据库维护两套商品定义。

### 5.2 页面布局

商城页改为响应式双区域布局。

- 宽屏：
  - 左侧：系统商城
  - 右侧：自定义奖励
- 窄屏：
  - 自动退化为上下堆叠，避免手机端左右过窄

### 5.3 系统商品行为

- 系统商品不可编辑。
- 系统商品不可删除。
- 系统商品允许兑换。
- 兑换后先进入背包。
- 用户在背包中点击“使用”后，生成只读奖励任务。

### 5.4 自定义奖励行为

- 保持现状：
  - 允许添加
  - 允许删除
- 创建时价格必须为正整数。
- 不支持免费自定义奖励。

### 5.5 价格规则

- 系统商品最低价格为 `1` 金币。
- 因此商城前端对购买行为的限制需要从“`cost <= 0` 禁止兑换”调整为“仅禁止负数价格商品”。
- 本次系统日常奖励均为 `effect_type = null`，复用现有“进背包后手动使用”的逻辑。

## 7. UI 设计说明

### 7.1 左侧系统商城

- 显示标题“系统商城”。
- 商品卡片展示：
  - 名称
  - 简短描述
  - 金币价格
  - 兑换按钮
- 不显示删除按钮。
- 不显示编辑入口。

### 7.2 右侧自定义奖励

- 显示标题“自定义奖励”。
- 保留现有添加按钮和删除按钮。
- 延续现有卡片结构与交互，减少破坏性变更。

### 7.3 响应式策略

- 采用 `LayoutBuilder` 或等价方式按宽度切换布局。
- 桌面与平板横向优先双栏。
- 手机与窄屏优先纵向堆叠。

## 8. 代码变更范围

预计涉及以下区域：

- Flutter
  - `frontend/lib/features/reward/screens/reward_shop_page.dart`
  - `frontend/lib/features/reward/controllers/reward_controller.dart`
  - `frontend/lib/core/i18n/app_locale_controller.dart`

  - `supabase/migrations/*` 新增一条系统日常商品种子数据迁移

  - `frontend/test/reward_controller_test.dart`
  - 新增商城页面/源码相关测试

## 9. 测试策略

遵循先测后改。

### 9.1 控制器测试

- 系统商品和自定义奖励能正确分组。
- `1` 金币系统商品允许兑换。
- 系统商品仍能复用现有购买链路写入背包。

### 9.2 页面测试

- 商城页面包含左右两块区域。
- 系统商城区域不出现删除入口。
- 自定义奖励区域保留删除入口与添加入口。
- 窄屏布局可回退为上下布局。

### 9.3 回归测试

- 背包页面使用奖励后仍可生成只读奖励任务。
- 现有 reward model / controller 测试继续通过。

## 10. 验收标准

- 用户能在商城页看到“系统商城”和“自定义奖励”两个区域。
- 左侧系统商城能展示内置初始商品。
- 系统商品不可编辑、不可删除。
- 系统商品可用金币兑换，并先进入背包。
- 背包中使用系统日常奖励后，会生成只读奖励任务。
- 右侧自定义奖励的添加和删除行为不被破坏。
- 所有新增与相关回归测试通过。

## 11. 风险与注意事项

- 当前前端购买限制若仍然拒绝 `cost <= 0`，会阻断低价商品策略；本次虽然最低价为 `1`，仍应把逻辑收敛为只拒绝非法负值，以免后续再次踩到边界。
- 数据迁移需要避免误删用户自定义商品。
- 旧系统主题/特效商品与本次新增的日常奖励商品会同时存在，应保证列表渲染清晰，不互相干扰。
